# Feature Comparison: app-store-scraper (TS) vs app-store-scraper-js

> Generated 2026-03-27

## Summary

(generated by Claude)

Both libraries provide the same core set of 10 scraping methods. The main differences are in **which Apple endpoints** they hit and a **memoization/caching** feature present in the JS repo but absent from the TS repo. Excluding rate limiting (out of scope), the TS repo is at feature parity or ahead on most fronts.

**Key finding:** Apple has removed the embedded bearer token from app page HTML. This means the `amp-api-edge` approach for privacy and version history is **permanently broken** in the JS repo. Our TS repo's HTML-scraping approach for these endpoints is the more resilient strategy.

## Functionality Missing from TS Repo

### 1. Memoization / In-Memory Caching

The JS repo exposes a `memoized(opts)` factory that returns a cached version of every method:

```js
const store = require('app-store-scraper').memoized({
  maxAge: 1000 * 60 * 5,  // 5 min TTL
  max: 1000                // max 1k cached results per method
});
// store.app(), store.search(), etc. are all cached
```

The TS repo has **no built-in caching**. Users must implement their own.

**Recommendation:** Consider adding an optional caching layer or documenting a pattern for external caching.

## API Endpoint Differences

These are not missing features, but the repos use different Apple endpoints for the same task. This matters because endpoint availability, response format, and data richness may differ.

### Search

| | TS Repo | JS Repo |
|---|---------|---------|
| **URL** | `https://itunes.apple.com/search` | `https://search.itunes.apple.com/WebObjects/MZStore.woa/wa/search` |
| **Auth** | None (public API) | `X-Apple-Store-Front` header required |
| **Response** | Standard iTunes JSON (`results[]`) | Internal format (`bubbles[0].results[]`) |
| **Device filter** | ✅ via `entity` param | ❌ (TODO in source) |
| **Pagination** | Client-side slice (API limit 200) | Client-side slice (plucks IDs → lookup) |

The JS repo uses Apple's internal storefront search, which may rank results differently. It also does an extra lookup call to hydrate full app objects from the returned IDs, while the TS repo gets full objects directly from the public API.

### List (Top Charts / Collections)

| | TS Repo | JS Repo |
|---|---------|---------|
| **URL** | `https://itunes.apple.com/{country}/rss/{collection}/...` | `http://ax.itunes.apple.com/WebObjects/MZStoreServices.woa/ws/RSS/{collection}/...` |
| **Protocol** | HTTPS | HTTP |
| **Store param** | Country in URL path | `?s={storeId}` query param |

Both return the same RSS JSON feed format. The JS endpoint uses an older Apple internal domain (`ax.itunes.apple.com`) over plain HTTP.

### Privacy

| | TS Repo | JS Repo |
|---|---------|---------|
| **Approach** | Scrapes HTML from `apps.apple.com` with Cheerio | Two-step: extracts bearer token from page, then calls `amp-api-edge.apps.apple.com` API |
| **URL** | `https://apps.apple.com/{country}/app/id{id}` | Step 1: same page for token; Step 2: `https://amp-api-edge.apps.apple.com/v1/catalog/{country}/apps/{id}?platform=web&fields=privacyDetails` |
| **Data richness** | Privacy policy URL + privacy type labels | Structured JSON with `privacyTypes[]`, each containing `dataCategories[]` and `dataTypes[]` |
| **Fragility** | Depends on HTML structure (CSS selectors) | Depends on token extraction regex + API stability |

**The JS approach would return more structured, granular privacy data** (categories, data types, purposes), but as of 2026-03-27, the bearer token extraction is broken and this endpoint **no longer works**. The TS HTML-scraping approach remains functional.

### Version History

| | TS Repo | JS Repo |
|---|---------|---------|
| **Approach** | Scrapes HTML from `apps.apple.com` with Cheerio | Two-step: extracts bearer token, then calls `amp-api-edge.apps.apple.com` API |
| **URL** | `https://apps.apple.com/{country}/app/id{id}` | Step 1: same page for token; Step 2: `https://amp-api-edge.apps.apple.com/v1/catalog/{country}/apps/{id}?platform=web&extend=versionHistory&additionalPlatforms=appletv,ipad,iphone,mac,realityDevice` |

**The JS approach could potentially retrieve version history across multiple platforms**, but as of 2026-03-27, the bearer token extraction is broken and this endpoint **no longer works**. The TS HTML-scraping approach remains functional.
